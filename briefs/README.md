# Content Brief Schema

A **content brief** (`content_brief.json`) is the per-application JSON file that drives `generate.js`. It tells the pipeline what to write, how to frame it, and which projects to surface — tailored to a specific company and role.

## How a brief is created

1. Run `brainstorm.sh` — it asks you a series of questions about the role and outputs a raw notes file.
2. Paste the notes (plus the job description) into an AI chat and ask it to produce a `content_brief.json` matching this schema.
3. Save the result as `briefs/[company-slug].json`.
4. Run `generate.js briefs/[company-slug].json` to produce the HTML output.

Briefs are **gitignored** — they contain personal career data and are never committed. See the note at the bottom of this file.

## `_meta` — brief metadata

| Field | Type | Required | Description |
|---|---|---|---|
| `schema_version` | string | required | Semver string for the brief schema (e.g. `"2.1.0"`). Used to detect breaking changes. |
| `generated` | string (date) | required | ISO date the brief was generated (`"YYYY-MM-DD"`). |
| `company` | string | required | Company display name (e.g. `"Acme Corp"`). Used in nav badge and page title. |
| `role` | string | required | Role title as it appears in the JD. |
| `jd_url` | string (url) | optional | URL to the original job posting. Used for internal reference only — not rendered. |
| `confidence_score` | number | optional | AI-assigned fit score (0–100) based on JD-to-profile match. Informational only. |
| `template_id` | string | required | Layout template to use. Currently only `"three-act"` is supported. |
| `design_system` | string | required | `"own"` to use the default FORA design system, or `"company"` to attempt a company-matched theme. |
| `company_ds_url` | string \| null | optional | URL to the company's public design system or Storybook (used if `design_system` is `"company"`). |
| `company_logo_url` | string \| null | optional | URL to the company logo (Clearbit format works well). Rendered in the nav. |
| `company_primary_color` | string \| null | optional | Hex color for accent theming (e.g. `"#FF5A1F"`). Falls back to default if null. |
| `posthog_event_tag` | string | optional | Slug used to tag PostHog analytics events for this brief (e.g. `"acme-senior-product-designer"`). |

## `opportunity_model` — AI analysis of the role

These fields are generated by the AI during brief creation. They inform tone and positioning choices but are not directly rendered.

| Field | Type | Description |
|---|---|---|
| `company_stage` | string | Free-text assessment of company stage and what that implies for the role. |
| `design_maturity` | string | How mature design is at this company, inferred from the JD. |
| `hiring_signal` | string | What the hiring signal tells you — backfill, expansion, first hire, etc. |
| `cultural_tone` | string | Cultural register inferred from JD language (e.g. builder-first, process-heavy). |
| `recommended_positioning` | string | The single positioning angle the AI recommends based on the above. |

## `static_wrapper` — page chrome

| Field | Type | Required | Description |
|---|---|---|
| `nav_badge` | string | required | Short label shown in the nav (e.g. `"For Acme"`). |
| `nav_portfolio_url` | string | optional | URL for the portfolio link in the nav. Falls back to a default if omitted. |

## `act1_who_i_am` — the opening act

| Field | Type | Required | Description |
|---|---|---|
| `emphasis` | string | required | Positioning archetype — e.g. `"builder"`, `"systems"`, `"research"`. Controls which variant of the intro block is rendered. |
| `positioning_line` | string | required | The one-line hook at the top of Act 1. Should be tailored to the role. |
| `philosophy_note` | string | required | A short, punchy belief statement. Rendered below the positioning line. |
| `philosophy_note_source` | string | optional | Internal tag describing where this line came from (e.g. `"core_belief"`, `"adapted"`). Not rendered. |

---

## `act2_what_ive_done` — the work

### Top-level fields

| Field | Type | Required | Description |
|---|---|---|
| `intro_line` | string | required | One-line framing for the work section (e.g. `"Three projects where the constraint made the work better."`). |
| `works` | array | required | Ordered list of work items. See below. |

### `works[]` — individual work items

| Field | Type | Required | Description |
|---|---|---|
| `order` | number | required | Display order (1-indexed). |
| `source_id` | string | required | Slug linking this item to a case study or portfolio source file. |
| `title` | string | required | Display title for the work item. |
| `framing_angle` | string | required | Why this project is relevant to this specific role. Written for the AI context — may inform generated copy. |
| `decision_to_surface` | string | required | The single most important decision to highlight. |
| `outcome_to_surface` | string | required | The result or impact to lead with. |
| `section_format` | string | required | Render format: `"case_study_link"`, `"signal_card"`, or `"timeline_entry"`. |
| `nda_note` | string \| null | optional | If present, instructs the renderer how to handle NDA constraints (what to show, what to omit). |
| `media` | object \| null | optional | Media attachment. See below. |

### `works[].media`

| Field | Type | Description |
|---|---|---|
| `type` | string | Media type: `"image"` or `"loom"`. |
| `file` | string \| null | Relative path to a local image file (used when `type` is `"image"`). |
| `url` | string \| null | URL to a Loom video or remote asset (used when `type` is `"loom"`). |
| `alt` | string | Alt text for accessibility. |
| `caption` | string | Caption rendered below the media. |

## `act3_what_ill_bring` — the 90-day plan

### Top-level fields

| Field | Type | Required | Description |
|---|---|---|
| `intro_line` | string | required | One-line framing for the 90-day section. |
| `day_15` | object | required | Day 15 milestone. See below. |
| `day_30` | object | required | Day 30 milestone. |
| `day_90` | object | required | Day 90 milestone. |

### `day_15` / `day_30` / `day_90` — milestone objects

| Field | Type | Required | Description |
|---|---|---|
| `title` | string | required | Short milestone label (e.g. `"Map the terrain"`). |
| `body` | string | required | 2–4 sentence description of what happens in this window. Rendered as body copy. |
| `credibility_anchor` | string | optional | A past project or experience that proves you can do this. Rendered as a footnote or annotation. |

---

## `cold_message` — outreach copy

| Field | Type | Description |
|---|---|---|
| `hook` | string | Opening line of the cold message. Should be specific, not generic. |
| `body` | string | 1–2 sentence bridge to the link. |
| `sign_off` | string | Name to sign off with. |

---

## `tone_notes`

A free-text string. Instructions for the AI (or for you) about register, vocabulary, and what to avoid when writing copy for this brief.

---

## Notes

**Briefs are gitignored.** The `briefs/` directory (except `example-brief.json`) is listed in `.gitignore`. Briefs contain personal career data, job-specific positioning, and potentially NDA-sensitive framing. They are never committed to the repo.

**`example-brief.json` is fictional.** It exists as a committed schema reference only. The company (`Acme Corp`), projects, and all content are made up. The file is never read by `generate.js` — it is documentation only.
